# EcomfyApp Mini Lead Routing Engine - Technical Documentation

## Stack Selection

**Python + FastAPI + SQLite** was chosen for the following reasons:

- **FastAPI** over Express/Flask: built-in request validation via Pydantic, automatic Swagger documentation, native async support for simulating buyer webhook timeouts, and type safety throughout the codebase.
- **SQLAlchemy ORM** over raw SQL: provides a clean data model layer, relationship management, and makes switching from SQLite to PostgreSQL a one-line configuration change.
- **SQLite** for the MVP: zero infrastructure setup, runs locally without Docker, and the schema is fully portable to PostgreSQL for production.
- **Pydantic** handles input validation (email format, phone format) at the API boundary, keeping service logic clean.
- **OpenAI API** for the optional AI summary: receives pre-calculated metrics and generates an executive report. The AI does not make routing decisions or generate data.

This stack prioritizes fast local setup, testability, and clear separation of concerns while remaining production-scalable.

---

## Architecture

The application is built as a **FastAPI monolith** backed by **SQLite** via **SQLAlchemy ORM**.

It follows a **service layer pattern** with clearly separated responsibilities:

- **Validation Service** - Validates incoming lead data (required fields, format, duplicates).
- **Routing Service** - Evaluates buyers against eligibility criteria and selects the best match.
- **Delivery Service** - Simulates ping/post delivery to buyers via webhooks.
- **Ledger Service** - Manages buyer balances, debits, credits, and refunds.
- **Alert Service** - Generates operational alerts and optionally sends them to Slack.
- **AI Summary Service** - Optionally generates natural-language daily summaries via OpenAI.

**Dependency injection** is handled through FastAPI's `Depends` mechanism, providing database sessions and service instances to route handlers.

## Data Model

The system uses 5 core tables:

### leads

Stores every incoming lead request.

| Field | Description |
|-------|-------------|
| id | Primary key |
| first_name, last_name | Contact name |
| email, phone | Contact information |
| state | US state code |
| vertical | Lead vertical (e.g., insurance, solar) |
| source | Traffic source identifier |
| status | Current status: `new`, `sold`, `unsold`, `returned` |
| buyer_id | Foreign key to the buyer who purchased the lead (nullable) |
| created_at | Timestamp of lead creation |

### buyers

Stores buyer configurations and current state.

| Field | Description |
|-------|-------------|
| id | Primary key |
| name | Buyer display name |
| status | `active` or `inactive` |
| campaign_status | `active` or `paused` |
| ping_tree_enabled | Whether buyer participates in ping tree |
| balance | Current account balance |
| price_per_lead | Cost charged per accepted lead |
| daily_cap | Maximum leads per day |
| daily_accepted | Counter of leads accepted today |
| accepted_states | JSON list of US states the buyer accepts |
| accepted_verticals | JSON list of verticals the buyer accepts |
| schedule_start, schedule_end | Operating hours (HH:MM format) |
| priority | Ping tree priority (lower = higher priority) |
| webhook_url | Simulated delivery endpoint |

### delivery_attempts

Records every ping/post attempt to a buyer for a given lead.

| Field | Description |
|-------|-------------|
| id | Primary key |
| lead_id | Foreign key to leads |
| buyer_id | Foreign key to buyers |
| status | `accepted`, `rejected`, `timeout`, `error` |
| reason | Rejection or error reason |
| response_time_ms | Simulated response time |
| created_at | Timestamp of the attempt |

### ledger

Immutable financial audit trail for all balance changes.

| Field | Description |
|-------|-------------|
| id | Primary key |
| buyer_id | Foreign key to buyers |
| lead_id | Foreign key to leads |
| type | Transaction type: `debit`, `credit`, `refund` |
| amount | Transaction amount |
| balance_before | Buyer balance before transaction |
| balance_after | Buyer balance after transaction |
| created_at | Timestamp of the transaction |

### alerts

Stores operational alerts generated by the system.

| Field | Description |
|-------|-------------|
| id | Primary key |
| type | Alert type (e.g., `low_balance`, `cap_reached`, `high_rejection`) |
| message | Human-readable alert description |
| buyer_id | Related buyer (nullable) |
| created_at | Timestamp |

### Relationships

- A **lead** belongs to zero or one **buyer** (via `buyer_id`).
- A **lead** has many **delivery_attempts**.
- A **buyer** has many **delivery_attempts**, **ledger** entries, and **alerts**.
- **Ledger** entries reference both the **buyer** and the **lead** involved.

## Routing Logic

When a lead is submitted, the routing service evaluates every buyer against **8 eligibility criteria**, checked in this order:

1. **Status** - Buyer must be `active`.
2. **Campaign Status** - Buyer's campaign must be `active` (not paused).
3. **Ping Tree** - Buyer must have `ping_tree_enabled = true`.
4. **Balance** - Buyer's balance must be >= their `price_per_lead`.
5. **Daily Cap** - Buyer's `daily_accepted` must be < `daily_cap`.
6. **State** - Lead's state must be in buyer's `accepted_states` list.
7. **Vertical** - Lead's vertical must be in buyer's `accepted_verticals` list.
8. **Schedule** - Current time must fall within buyer's `schedule_start` and `schedule_end`.

Each buyer receives a **BuyerEvaluation** object containing:
- `buyer_id` and `buyer_name`
- `eligible` (true/false)
- `reason` (why they were excluded, if ineligible)

Eligible buyers are **sorted by priority** (ascending - lower number = higher priority) before delivery is attempted.

Every evaluation decision is **traceable** and can be inspected for debugging and auditing.

---

## Ping Tree and Fallback

The ping tree processes eligible buyers in **priority order**:

1. The system takes the sorted list of eligible buyers.
2. For each buyer, a **delivery attempt** is made (simulated webhook call).
3. Each attempt is recorded in the `delivery_attempts` table with its status and response time.
4. If the buyer **accepts**, the lead is marked as `sold`, the buyer is debited, and routing stops.
5. If the buyer **rejects** or **times out**, the attempt is recorded and the system moves to the **next buyer**.
6. If **all buyers fail** (reject, timeout, or error), the lead is marked as `unsold`.

This ensures maximum fill rate by exhausting all eligible buyers before giving up.

## Error Handling

The system handles errors at multiple levels:

- **Pydantic validation** catches malformed requests and returns `422 Unprocessable Entity` with detailed field-level errors.
- **Business validation** (duplicate detection, missing fields) returns structured rejection reasons without raising HTTP exceptions.
- **HTTP exceptions** are raised for invalid operations:
  - `404 Not Found` for non-existent leads or buyers.
  - `400 Bad Request` for invalid state transitions (e.g., returning an unsold lead).
- **Slack failures** are caught and logged but **never block** lead processing. If the Slack webhook is unreachable, the alert is still saved to the database.
- **AI summary failures** are caught and logged but **never block** report generation. The report returns normally with a fallback message instead of the AI summary.

## Duplicate Prevention

Duplicates are prevented at **two layers**:

### System-Level Dedup

During the validation stage, the system checks for any existing lead with the **same phone or email** submitted within the last **24 hours**. If a match is found, the lead is immediately rejected with a duplicate reason before routing begins.

### Buyer-Level Dedup

Buyer B is configured to simulate buyer-side duplicate detection. When a delivery attempt is made to Buyer B, it checks its own history of accepted leads. If the lead's phone or email matches a previously accepted lead, Buyer B rejects the delivery with a duplicate reason.

These two layers work together: system-level dedup catches obvious duplicates early, while buyer-level dedup handles cases where the same contact submits through different sources or with slight variations.

---

## Timeout Handling

Buyer C is configured to simulate a **webhook timeout**:

- When delivery is attempted to Buyer C, the system introduces a short delay to simulate an unresponsive endpoint.
- In tests, this delay is **0.1 seconds** to keep the test suite fast.
- The timeout threshold is configurable via the `BUYER_TIMEOUT_SECONDS` environment variable.
- When a timeout occurs, the system:
  1. Records the delivery attempt with `status = timeout`.
  2. Moves to the next buyer in the ping tree.
  3. If no more buyers are available, the lead is marked as `unsold`.

---

## Double-Charge Prevention

The system prevents double-charging through several mechanisms:

1. **Pre-delivery balance check** - During buyer evaluation (before any delivery attempt), the system verifies the buyer's balance is sufficient. Buyers with insufficient balance are excluded from the eligible list entirely.

2. **Post-acceptance debit only** - The buyer's balance is only debited **after** the buyer confirms acceptance. No charge occurs on rejection, timeout, or error.

3. **Audit trail** - Every ledger entry records `balance_before` and `balance_after`, providing a complete audit trail for financial reconciliation.

4. **Atomic transactions** - The balance debit and lead status update (`sold`) happen within the **same database transaction**. If either operation fails, both are rolled back, preventing inconsistent state.

## Scaling to Production

While the MVP uses SQLite for simplicity, the architecture is designed for straightforward scaling:

### Database
- **Replace SQLite with PostgreSQL** by changing the `DATABASE_URL` environment variable. SQLAlchemy abstracts the database engine, so no code changes are needed.
- **Add connection pooling** via SQLAlchemy's built-in pool settings (`pool_size`, `max_overflow`, `pool_timeout`).

### Application
- **Use async SQLAlchemy + async FastAPI** for handling concurrent requests without blocking threads.
- **Add Redis** for caching buyer configurations, reducing database reads on hot paths.
- **Horizontal scaling** with multiple `uvicorn` workers behind **nginx** as a reverse proxy and load balancer.

### Operations
- **Add Alembic** for database migrations to manage schema changes safely.
- **Implement rate limiting** per source to prevent abuse and ensure fair resource allocation.
- **Add a webhook queue** (Celery + Redis) for actual buyer delivery, decoupling the HTTP response from the delivery process and enabling retries.

## Integration with Phonexa, Everflow, GHL, Stripe

### Phonexa

Replace the simulated delivery layer with **Phonexa's ping/post API**:

- Map each `buyer_id` in the system to a corresponding **Phonexa campaign ID**.
- Use Phonexa's **real-time bidding** mechanism for buyer selection instead of the local ping tree.
- Receive acceptance/rejection responses from Phonexa and record them in `delivery_attempts`.
- Leverage Phonexa's built-in duplicate detection and compliance features alongside the system's own checks.

### Everflow

Use **Everflow's offer/conversion API** for tracking and attribution:

- Map leads to **Everflow offers** based on vertical and buyer configuration.
- Track **conversions and revenue** through Everflow's postback system.
- Use Everflow's reporting dashboard for cross-platform performance analysis.
- Implement Everflow's click/conversion tracking to measure source quality.

### GoHighLevel (GHL)

Push accepted leads to **GHL** for CRM and automation:

- Use GHL's REST API to **create or update contacts** with the lead's data upon acceptance.
- **Trigger GHL workflows/automations** on lead acceptance (e.g., SMS follow-up, email sequences, appointment booking).
- Sync lead status changes (returns, updates) back to GHL to keep records consistent.
- Use GHL's pipeline features to track lead progression post-sale.

### Stripe

Replace the internal balance tracking system with **Stripe billing**:

- **Create Stripe customers** for each buyer in the system.
- Use **Stripe Invoicing** to charge buyers for accepted leads, replacing the internal ledger.
- Implement **Stripe webhooks** to listen for payment confirmation before delivering the lead to the buyer.
- Handle failed payments by holding lead delivery until payment is confirmed.
- Use Stripe's reporting and dashboard for financial reconciliation instead of the internal ledger reports.
